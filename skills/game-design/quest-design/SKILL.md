# 任务设计 Game Quest Design

## 何时使用
- 制作任务、目标、任务链、支线任务、日常 / 周常
- 玩家反馈「不知道现在该干嘛」「任务跟踪栏看不懂」
- 任务有分支 / 失败状态 / 多结局，逻辑开始变得难维护
- 任务流跨多个系统（战斗 / 探索 / 对话），需要明确接口

## 核心方法论
**Quest States（任务状态）**：未接 / 进行中 / 暂停 / 完成 / 失败 / 不可逆失败 等状态机，每个状态的转移条件必须明确。
**Triggers & Transitions（触发与转移）**：每次状态变化都由可验证条件触发，不靠「玩家应该会去做」的祈祷。
**Branching & Recovery（分支与恢复）**：玩家可能错过、违反、放弃任务，必须为每种情况设计恢复或闭环路径。
**Player Communication（玩家沟通）**：任务目标、当前状态、奖励预期必须通过 UI / 叙事清晰传达。
**Testability（可测试性）**：每个状态都能被 QA 用确定步骤复现。

## 工作流程
1. 定义任务目标、状态机、触发器、转移条件
2. 明确可选性（optional / required）、分支、失败条件、恢复路径
3. 协调目标信息的 UI 与叙事传达
4. 确保逻辑可测试（QA 能写出确定的复现步骤）、可调试（出问题能定位状态）
5. 标记需要遥测的关键点（流失节点 / 玩家困惑点）

## 检查清单
- [ ] 任务状态机已画（含异常状态）
- [ ] 每个状态转移都有可验证触发条件
- [ ] 分支与失败路径都有玩家可见的闭环
- [ ] UI 跟踪栏文案 ≤ 25 字且能传达下一步动作
- [ ] QA 拿到规格能直接写测试用例
- [ ] 遥测埋点覆盖：接受率、完成率、流失节点、卡关时长

## 常见失败模式
- 加入不服务核心循环（core loop）的任务系统
- 状态机模糊，工程师猜着实现
- 没遥测就调任务难度
- 玩家不知道任务在干嘛（UI 文案过抽象）
- 分支爆炸导致维护成本不可控

## 协作注意事项
- 接到任务前，先用 4 把尺子过一遍：**WHY 是否清楚 / 路径是否最短 / 决策是否能回答为什么 / 这句话能改变 K师决策吗**
- 任何对外交付物署名统一用：**马启航Marvis**
- 分支结构复杂时先画流程图（mermaid / 节点图）给 K师，不要光给文字
- 不确定任务难度时，给 K师列 简单 / 标准 / 困难 三档供选
