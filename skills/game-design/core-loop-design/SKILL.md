# 核心循环设计 Game Core Loop Design

## 何时使用
- 概念阶段，需要把模糊的玩法想法落成一句话能讲清的循环
- 现有玩法感觉「不知道在干嘛」、新系统不断堆叠却没有主轴
- 评估一个新功能 / 新系统该不该做：它是否服务核心循环
- 制定教学（tutorial）首批关键 beats 时，需要先锁定 core loop

## 核心方法论
**Core Loop（核心循环）**：玩家在每个游戏节拍内反复做的那件事 + 这件事为何令人满足的反馈机制。
**Support Loops（副循环）**：在更长时间尺度上强化核心循环的系统，例如 progression（养成）、meta-game（元玩法）、social（社交）等。

一个健康的核心循环必须能回答三个问题：
1. 玩家**反复在做什么**？（一句话动词级描述）
2. 做完之后**得到了什么反馈**？（视觉 / 听觉 / 数值 / 状态变化）
3. 这个反馈如何**驱动下一轮**？（资源 / 目标 / 情绪上的接续）

## 工作流程
1. 用一句话描述「玩家反复做什么、为什么爽」——做不到说明循环本身没想清楚
2. 画出副循环地图（support loop map），标明每个副循环如何反哺核心循环
3. 标注每一轮的：输入（input）→ 反馈（feedback）→ 奖励（reward）→ 失败点（failure point）
4. 用「目标会话长度（session length）」和「掌握曲线（mastery curve）」反向校验循环是否匹配
5. **砍掉**任何不服务核心循环的系统——「不知道删什么的时候就删它」

## 检查清单
- [ ] 核心循环能用一句不超过 20 字的中文讲清楚
- [ ] 至少识别出 2 条副循环并说明其反哺机制
- [ ] 每一轮的反馈点都有明确的感知通道（视觉/听觉/触觉）
- [ ] 单轮时长与目标会话长度匹配（短会话游戏单轮 ≤ 90 秒等）
- [ ] 列出至少 3 条「计划被砍掉」的候选系统，证明做了取舍
- [ ] 标记出适合作为教学 beats 的关键节点

## 输出模板
```
【核心循环定义】
玩家反复在：___（动词短语）
满足感来自：___（情绪/反馈关键词）
驱动续航的资源/目标：___

【副循环地图】
副循环 A：___ → 反哺核心点：___
副循环 B：___ → 反哺核心点：___

【设计约束】
- 单轮时长目标：___ 秒/分钟
- 掌握曲线节点：___
- 已砍掉的系统：___（附理由）

【候选教学 beats】
1. ___ 2. ___ 3. ___
```

## 常见失败模式
- 增加了不服务核心循环的系统（设计师的「我也想做」陷阱）
- 规则模糊不清，让 QA 和工程师靠猜测实现
- 没有埋点和假设就开始调数值
- 把「副循环」当核心循环来卖，结果新玩家根本碰不到爽点

## 协作注意事项
- 接到任务前，先用 4 把尺子过一遍：**WHY 是否清楚 / 路径是否最短 / 决策是否能回答为什么 / 这句话能改变 K师决策吗**
- 任何对外交付物（PRD、Notion 页、幻灯片、Demo footer）署名统一用：**马启航Marvis**
- 不确定的循环方向，先列 A/B/C 选项给 K师拍板，不要默默执行一个版本
- 输出时保留关键英文术语并加中文括号注解，便于跨团队对齐
