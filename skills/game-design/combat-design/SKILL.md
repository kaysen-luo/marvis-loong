# 战斗系统设计 Game Combat Design

## 何时使用
- 立项或迭代一款战斗驱动的游戏（动作、射击、Roguelike、ARPG 等）
- 战斗体验「看不懂」（readability 差）或「太浅」（缺乏 skill expression）
- 数值调整反复横跳，缺一套可解释的平衡框架
- 准备给战斗加新武器、新敌人、新机制前

## 核心方法论
**Player Verbs（玩家动作动词）**：玩家可以做的核心动作集合，例如 攻击 / 闪避 / 防御 / 招架 / 位移 / 反制（counter）。
**Enemy Pressure（敌人压力）**：敌人通过哪些手段持续给玩家施压（攻速、追击、范围、组合），以及对应的应对手段。
**Resource Economy（资源经济）**：耐力、子弹、技能 CD、血量等在战斗中的进出与权衡。
**Readability（可读性）**：玩家能否在 0.5 秒内识别威胁类型、判断该用哪个 verb 应对。
**Tuning Hooks（调参挂钩）**：可以单独拨动的参数集合，必须事先暴露出来。

## 工作流程
1. 定义玩家动作动词集合，明确每个 verb 的**反制对象**、**风险收益**（risk/reward）、**节奏占位**
2. 设计敌人角色矩阵（enemy roles）：远程压制 / 近战施压 / 控场 / 精英威胁等，明确每个角色的压力模式
3. 指定**反馈层**（视觉、音效、震动、屏幕特效）、资源使用、失败状态
4. 列出可独立拨动的调参变量与遥测（telemetry）需求
5. 通过 playtest 验证：清晰度（clarity）/ 公平性（fairness）/ 技巧表达空间（skill expression）

## 检查清单
- [ ] 玩家 verbs 清单完整，每个都有明确反制对象
- [ ] 敌人角色矩阵覆盖至少 4 种压力模式
- [ ] 每个 verb 在画面中有 ≤ 0.3 秒内可识别的反馈
- [ ] 资源（耐力/弹药/CD）的进出曲线明确
- [ ] Tuning 变量表已列出，每个变量都标注「拨动它影响什么」
- [ ] Playtest 问题清单已准备（至少 5 条）

## 常见失败模式
- 增加了不服务核心循环（core loop）的战斗系统
- 规则模糊，QA / 工程师只能凭经验猜
- 没有埋点和假设就开始调数值
- Verbs 过多导致玩家选择瘫痪，或过少导致「一招鲜」

## 协作注意事项
- 接到任务前，先用 4 把尺子过一遍：**WHY 是否清楚 / 路径是否最短 / 决策是否能回答为什么 / 这句话能改变 K师决策吗**
- 任何对外交付物署名统一用：**马启航Marvis**
- 战斗手感很主观，给 K师拍板时务必带上 A/B 对比视频或 GIF，不要光给文字
- 调参建议附「为什么这样调」的一句话假设
