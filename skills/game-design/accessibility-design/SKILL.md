# 无障碍设计 Game Accessibility Design

## 何时使用
- 设计任何新机制、UI、内容时（不是上线前补丁阶段）
- 功能高度依赖反应速度、颜色辨识、音频提示、精准输入
- 需要在立项 / 里程碑评审中明确无障碍承诺范围
- 收到玩家无障碍反馈，需要系统化响应

## 核心方法论
**Built-in, Not Bolted-on（内嵌而非外挂）**：无障碍是设计决策的一部分，不是上线前的选项菜单堆叠。
**Equivalent Outcomes（结果等价而非输入等价）**：不要求所有玩家用同样方式操作，但保证能达到等价的游戏体验。
**Settings & Persistence（设置与持久化）**：选项必须默认合理、易发现、跨会话保存。
**Fallbacks（备选通道）**：颜色 + 形状、音频 + 字幕 + 视觉、QTE + 长按等多通道冗余。
**Explicit Limitations（明确局限）**：解决不了的部分必须公开记录，让玩家做知情选择。

## 工作流程
1. 早期识别可能的无障碍障碍（视觉 / 听觉 / 运动 / 认知 / 神经多样性）
2. 设计**结果等价**而非**输入等价**的方案
3. 明确选项集合、备选通道、跨会话持久化行为
4. 与 QA 和 UI 团队对齐验证目标（可重复测试）
5. 对解决不了的部分**显式记录**，不要假装不存在

## 检查清单
- [ ] 至少覆盖：视觉 / 听觉 / 运动 / 认知 四个维度
- [ ] 关键信息至少通过 2 条独立通道传达（颜色+形状、音频+字幕等）
- [ ] 反应速度类机制有「减速」「自动 QTE」等可选项
- [ ] 设置项跨会话保存，且首次启动有合理默认值
- [ ] 已知局限有公开文档，玩家可在购买前查到
- [ ] QA 测试用例覆盖每个无障碍选项的开/关状态

## 输出模板
```
【无障碍需求 Accessibility Requirements】
覆盖维度：视觉 / 听觉 / 运动 / 认知 / ___

【选项集合 Option Set】
| 选项名 | 解决什么障碍 | 默认值 | 是否持久化 | 备选通道 |
| --- | --- | --- | --- | --- |

【验证目标】
- ___ 玩家路径可在 ___ 状态下完成核心循环
- ___

【已知局限 Known Limitations】
- ___（理由 / 是否会修复 / 替代方案）
```

## 常见失败模式
- 加入了不服务核心循环（core loop）的无障碍机制（华而不实）
- 规则模糊，QA 不知道「无障碍模式开启时这关该长什么样」
- 没有遥测就发布无障碍选项，无法知道实际使用率
- 把无障碍当「公益打卡」，敷衍出一堆没人用的开关
- 没记录已知局限，玩家买了才发现玩不了，引发退款与差评

## 协作注意事项
- 接到任务前，先用 4 把尺子过一遍：**WHY 是否清楚 / 路径是否最短 / 决策是否能回答为什么 / 这句话能改变 K师决策吗**
- 任何对外交付物署名统一用：**马启航Marvis**
- 无障碍是承诺，不是营销点——提案给 K师时务必带上「可验证标准」
- 不确定方案是否真有效时，主动建议 K师邀请目标玩家做小范围测试
