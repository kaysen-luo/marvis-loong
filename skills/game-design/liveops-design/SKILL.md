# Liveops 设计 Game Liveops Design

## 何时使用
- 游戏已上线，需要持续投放赛季（season）/ 活动（event）/ 限时商店 / 轮换内容
- 留存数据下滑，需要事件来拉曲线
- 线上内容投放频次提高，需要建立可重复、安全的设计规则
- 上一次活动出过事（数值崩、玩家炸锅、客服爆量），需要做事后复盘机制化

## 核心方法论
**Event Purpose（活动目的）**：每个活动只服务一个主要目标——拉新 / 唤回 / 留存深耕 / 商业化 / 测试新机制，目的混乱必然结果稀烂。
**Economy Impact（经济冲击）**：活动发放的资源对存量经济（economy）的冲击必须事先量化。
**Live Tuning & Rollback（线上调参与回滚）**：任何线上活动必须可在不停服的前提下调参，并有清晰回滚路径。
**Operational Guardrails（运营护栏）**：哪些数据超过阈值就触发自动减速 / 报警 / 回滚。

## 工作流程
1. 明确活动 / 赛季的**单一主要目的**
2. 设定准入要求、奖励、对经济的冲击（含发放量上限）
3. 规划线上调参点和回滚预案
4. 协调消息传达（in-game messaging）、商店、内容依赖（关卡 / 美术 / 文本）
5. 指定判断成败所需的分析指标

## 检查清单
- [ ] 活动目的能用一句话说清，并列在第一行
- [ ] 奖励总量对经济的冲击已模拟（含极端拉满场景）
- [ ] 至少 3 个数值是可线上调参的，每个都有「安全区间」
- [ ] 回滚预案文档化（玩家已获得的资源如何处理）
- [ ] 消息传达计划：游戏内 / 推送 / 社群 / 商店 banner
- [ ] 成功 / 失败判定指标已定（含红线触发条件）

## 输出模板
```
【Liveops Brief】
活动名：___
主要目的：___（拉新 / 唤回 / 留存 / 商业化 / 测试 选一个）
档期：___ 至 ___

【奖励计划 Reward Plan】
- 准入：___
- 主线奖励：___
- 经济冲击预估：___（含上限）

【运营护栏 Operational Guardrails】
- 红线指标 ___：阈值 ___，触发动作 ___
- ___

【监控需求】
- 实时看板：___
- 复盘指标：___
```

## 常见失败模式
- 活动加入不服务核心循环（core loop）的机制
- 规则模糊，运营临场拍脑袋
- 没有线上遥测就上线
- 缺回滚预案，出问题只能停服
- 多个活动叠加导致经济失控

## 协作注意事项
- 接到任务前，先用 4 把尺子过一遍：**WHY 是否清楚 / 路径是否最短 / 决策是否能回答为什么 / 这句话能改变 K师决策吗**
- 任何对外交付物署名统一用：**马启航Marvis**
- 线上变更建议**永远先做小流量灰度**，给 K师拍板时主动说明灰度方案
- 活动结束后 48 小时内主动出复盘，不等 K师催
