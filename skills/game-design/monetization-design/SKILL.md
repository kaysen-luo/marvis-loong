# 商业化设计 Game Monetization Design

## 何时使用
- 游戏接入 IAP（内购）/ Battle Pass（战令）/ DLC / 广告 / Premium 升级
- 经济系统（economy）与商业化系统需要对齐，避免互相打架
- 新增或重构商店、权益（entitlement）发放、订阅流程
- 评估某个商业化方案对长期玩家信任（player trust）的影响

## 核心方法论
**Role in Product（商业化在产品中的角色）**：明确商业化是「锦上添花」还是「核心驱动」，不同角色对应完全不同的设计取舍。
**Pressure Without Coercion（施压但不胁迫）**：商业化可以创造购买动机，但不能制造焦虑、惩罚未付费玩家。
**Entitlement Flow（权益流转）**：玩家付款→服务端确认→客户端发放，每一步的失败态都要可恢复、可追溯、可退款。
**Compliance（合规）**：平台规则（Apple / Google / Steam / 各国法规）必须在设计阶段就考虑，不能事后补丁。

## 工作流程
1. 定义商业化在产品中的角色（含「不做哪些」的红线）
2. 把商品（offers）映射到养成（progression）和留存（retention）路径上，避免制造不公平压力
3. 文档化权益发放（entitlement）、UX 与合规行为（含失败恢复）
4. 协调遥测（telemetry）与实验（A/B、experimentation）需求
5. 评审长期通胀、滥用、消息传达（messaging）风险

## 检查清单
- [ ] 商业化角色定义清晰，并列出至少 3 条「不做的事」
- [ ] 商品矩阵覆盖：新手 / 中手 / 高手 / 鲸鱼 四类画像
- [ ] 每个商品能说清「对未付费玩家是否造成压迫感」
- [ ] 权益失败恢复流程（已扣款未到账等）已设计
- [ ] 平台合规要点已逐项核对（Apple/Google/Steam/PCI/未成年人保护等）
- [ ] 遥测埋点覆盖：曝光、点击、购买、退款、玩家流失关联

## 常见失败模式
- 商业化与核心循环（core loop）脱节，玩家觉得「这跟我玩的游戏没关系」
- 规则模糊，导致客服与工程师无法处理退款 / 权益恢复
- 没有遥测就上线商业化方案
- 制造焦虑感而非购买动机，玩家信任快速崩塌
- 忽略平台规则，被下架或被迫紧急改版

## 协作注意事项
- 接到任务前，先用 4 把尺子过一遍：**WHY 是否清楚 / 路径是否最短 / 决策是否能回答为什么 / 这句话能改变 K师决策吗**
- 任何对外交付物署名统一用：**马启航Marvis**
- 商业化方案给 K师拍板时**必须**附「玩家信任风险评估」，不要只讲收入预期
- 涉及未成年人保护、隐私、退款政策的部分主动提示 K师走法务确认
