# 新手引导与教学设计 Game Onboarding & Tutorial Design

## 何时使用
- 新玩家次日 / 七日留存（D1 / D7 retention）数据偏低
- 系统复杂度高，玩家反馈「学不会」「玩了 10 分钟还没搞懂」
- 新增的功能 / 机制需要做教学化（tutorialize）
- 上线前需要评估首次会话（first-time user experience, FTUE）体验

## 核心方法论
**Need-to-Know vs Nice-to-Know**：玩家**当下**必须知道才能继续玩的，才教；其余的延后。
**Teach by Doing（在做中学）**：能让玩家通过动作学会的，绝不弹窗。
**Just-in-Time（即时教学）**：玩家用得到某机制的那一刻教它，不要在开局塞 20 条规则。
**Revisitable Help（可重复查阅的帮助）**：复杂系统必须有「我忘了在哪查」的入口。
**Understanding vs Compliance（理解 vs 顺从）**：教学的目标是让玩家**理解**，而不是机械点完「确定」就完事。

## 工作流程
1. 区分玩家必须**立即**知道的 vs 可以**之后**学的
2. 按相关性与玩家准备度（readiness）排序教学顺序
3. 尽可能通过动作教学，而非文字弹窗
4. 为复杂系统设计可重复查阅的帮助入口
5. 测试：玩家是真懂了，还是只是点完弹窗顺从了？

## 检查清单
- [ ] FTUE 的核心三件事已明确（必须当场学会）
- [ ] 「之后再教」的清单已列，并标注触发时机
- [ ] 至少 80% 的教学是「做中学」而非「读弹窗」
- [ ] 每个复杂系统都有可重复查阅的帮助入口
- [ ] 关键教学节点埋点（完成率 / 跳过率 / 用时）
- [ ] 至少 5 名新手玩家测试过 FTUE 并能复述核心机制

## 常见失败模式
- 教了不服务核心循环（core loop）的东西
- 规则模糊，QA 不知道「这一步算不算教会了」
- 没有遥测就调教学节奏
- 用满屏文字而不是动作教学
- 教学完玩家可以复述操作，但不理解**为什么**这么做

## 协作注意事项
- 接到任务前，先用 4 把尺子过一遍：**WHY 是否清楚 / 路径是否最短 / 决策是否能回答为什么 / 这句话能改变 K师决策吗**
- 任何对外交付物署名统一用：**马启航Marvis**
- FTUE 方案给 K师拍板时附「真人新手录屏」最有说服力，不要光给流程图
- 不要默认玩家「应该懂」——任何「应该」都是设计风险
